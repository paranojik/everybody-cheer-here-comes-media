- We still can’t even reuse images. Embedding only works for images and even that without cropping or resizing.

- Of course we don’t. That’s why a group of people got together at DrupalCon prague in 2013. Janez Urevc held a core conversation was titled “LET’S FIX FILE AND MEDIA HANDLING ONCE AND FOR ALL”.
- A plan was drafted for generic media handling. Later next year at Drupal camp NYC the media initiative was formed and a huge effort has been undertaken - unify media handling and provide a generic and versatile base for all kinds of media solutions. This is how media_entity was born.
- Main goals: Media library, reuse of images, upload images to your server, remote media from services like youtube, flickr, Generic handling of media with embedding, cropping and metadata.

- The Media entity module provides a 'base' entity for storing media. This is a normal content entity (just like node) which can reference all kinds of media-objects (local files - meaning on your server, YouTube videos, Tweets, Instagram photos, ...). Media entity provides a relation between Drupal and the actual media resource. You can reference this entity within any other Drupal entity via the Entity reference field.
- Media introduces the notion of Media sources. Media sources are plugins that implement the actual retrieval of the linked media object and its metadata and propagate storage to Drupal fields.
- On youtube you have a video a description thumbnail… In the past it was hard to use that. There was no standardized way of fetching and exposing that data on the site. With media entity we now have metadata fields and you can map those from the Source to local fields.” Tweet author, id, retweet count,...

Media entity

  • Propagate storage to fields
  • Media sources provide metadata:
    • Thumbnail
    • Description
    • Size
  • Mapping of metadata to entity fields
- After the official media initiative was announced at DrupalCon New Orleans work has begun to bring the media_entity into core. It is now called only “Media”. It’s a stable but hidden module. It is hidden because it actually doesn’t do much on it’s own. So far it only provides support for local images and documents (pdf,...) and a lot of the usability stuff needs to be addressed. So it is still up to contrib to leverage the media APIs and extend the functionality just as before.
- Media_entity is committed to core as stable but because a lot of the contrib is not yet ported to the new API and the upgrade path is not ready yet it is a hidden module and a lot of the usability stuff like sensible defaults, formatters and widgets are not in place yet. Media is an API module and does nothing special on it’s own.
- Currently the media initiative is also working on porting the existing media_entity ecosystem to be compatible with the recent changes in core. If you’re a developer and you want to help with you can join the sprints which are happening here at DCon vienna on Friday.
- But let’s not underestimate the impact this has on the overall progress of Drupal. The task and ultimate goal of core is not only to provide functionality but most importantly to set standards and provide APIs that contrib can then follow along, extend that functionality and provide richer customized experiences.

Entity browser

Entity browser

  • Selecting from library
  • Creating a new item while referencing
  • Uploading multiple images at once
  • More complex workflows
The Entity Browser module provides a very flexible and powerful UI component for browsing and selecting entities.

Entity browser - architecture

Source: https://slashrsm.github.io/entity-browser-session/#/architecture-wireframe

- It has a pluggable architecture, meaning everybody can implement parts which best suit their needs.
- The main part is called the WIDGET. That’s where the entity selection is taking place. It could be a view or an upload form, a form connected to some external DAM system, etc…
- On top we have the “Widget selector” Part that defines how switching between widgets work. Can be dropdown, buttons, tabs,...
- At the bottom we have the “Selection display”: This part that displays entities that are currently selected. It’s like some sort of clipboard.

Entity browser in use

  • As an entity reference field widget
  • As a file/image field widget
  • With Entity embed
  • In a custom form (as a form element)
  • In combination with Inline Entity Form
    So possible use cases would be:
  • selecting images from a library
  • Uploading images in bulk
  • It even supports a very flexible multi-step workflow

Entity embed

"The Entity Embed module allows any Drupal entity to be embedded within a text area using a WYSIWYG editor."

- This is where the powerful combination of Entity Embed comes into play. Predicated on, and extending the foundation established by Drupal core, these two modules were designed to be a standardized solution for embedding any entity or remote URL content in a WYSIWYG editor in Drupal 8.

Embed anything

  • Video
  • Related article (node)
  • List of related articles (Taxonomy)
  • Related product (Commerce product)
  • Discussion thread (Comment)
  • ...
- The module doesn't care what you embed, as long as that piece of content is stored in Drupal as a standard entity. In other words, you can embed nodes, taxonomy terms, comments, users, profiles, forms, media entities, etc. all in the same way and using a standard workflow.
Let’s see it in action. 1.) After downloading and installing the Entity Embed module and its dependencies, navigate to Configuration -> Content Authoring -> Text editor embed buttons and click on Add embed button
2.) On this page you will see that there is already one embed button created - this is used to embed nodes and is automatically created upon install.
3.) In our example we’re creating a button to embed images and videos, let’s call it Multimedia.
4.) We will be using an entity browser to upload and reuse images, so we select that one.
5.) As a tip, we’d always suggest you add an icon to your embed buttons in order to distinguish them. If you don’t do that, the icon will be set by default to the E icon and might be confusing for the user when having different embed buttons in the toolbar.
6.) Now let’s navigate to the text editors screen and add our newly created button to the toolbar.

Image cropping

And that is... how we handle image cropping.

Crop API

  • Storage for all cropping solutions
  • Handling crops based on:
    • Region - storing a crop area
    • Reference point
- The Crop API provides storage for “hopefully” all cropping solutions. It is able to handle crops based on region and reference point concepts. We won’t get into details, but rather show two different implementations based on the Crop API.
- Cropping information is saved per file. The problem is that the image style system does not have knowledge about the usage of the file.

Image widget crop

  • Specify exact crop region
  • Manual - It’s a field widget
  • A lot of manual work if we have to define many crops
Provides an interface for using the features of the Crop API. This element provides an UX for using a crop on all fields images or file elements. This module has particularity to purpose capability to crop the same image by “Crop type” configured. It’s very useful for editorial sites or media management sites.

Focal point

  • Crop around a selected point
  • Semi-Automatic
With focal point you want to say: this is the part of the image that has the most information - the part of the image I really want to be the center of the cropped version. And drupal then makes sure that that part of the image remains visible in the cropped version.
- In this short demo we’re focusing on the most important part of the image, first with the image widget crop, where we select an area which we want to crop.
- In the second example we are just selecting a point where we want the image to be focused.
